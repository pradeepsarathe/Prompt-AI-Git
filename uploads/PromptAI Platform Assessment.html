Platform assessment

PromptAI — end-to-end review & recommendations

Reviewed against pradeepsarathe/Prompt-AI-Git@main (source of truth) and the live deployment at promptai.in · June 11, 2026 · Perspectives: software architect, product owner, end user

What was reviewed

All three pages (briefing homepage, Learn AI, Archive), the shared front-end layers (pai-feed-engine.js, pai-google-ui.js, pai-account.js, pai-translate.js, pai-chrome.*), all nine Cloudflare Pages Functions (auth, subscribe, unsubscribe, send-digest, stats, feed, rss-proxy, sitemap, archive-data), SEO infrastructure (robots.txt, sitemaps, structured data, _redirects), and the email pipeline (Resend, welcome + digest templates).

Executive summary

PromptAI is a genuinely well-crafted, zero-cost AI briefing site. The engineering shows real maturity for a static-hosted product: layered RSS fallback chains, quality/spam filters, source-health tracking, localStorage caching, KV-backed shared counters, honeypot + rate-limited signup, one-click List-Unsubscribe headers, and graceful offline fallbacks nearly everywhere. The new briefing design is clean, fast-feeling, and coherent across all three pages.

Three structural ceilings now cap everything else:

  • 1. All content is assembled in the visitor's browser. Every visitor's device fans out to ~20 RSS feeds and up to 60 Hacker News item fetches, partly through free third-party proxies. This caps performance, reliability, consistency, privacy and — most importantly — SEO: Google indexes three page shells, not a single story.
  • 2. The brand promises AI; the platform contains none. The welcome email promises "one paper, explained in plain English," but every item is a raw RSS description. There is no summarization, no explanation, no prompts — despite the name PromptAI.
  • 3. Trust-layer basics are missing. No privacy policy or terms while collecting emails and passwords; no password reset; no login rate-limiting; no double opt-in; email auth (SPF/DKIM/DMARC) unverified.

The good news: ceiling #1 has a single, well-scoped fix (a scheduled aggregation Worker) that simultaneously improves performance, SEO, reliability and architecture — it is the highest-leverage change in this report (R1).

1 · Architecture & technical debt

Software-architect perspective
R1

Move feed aggregation server-side (the keystone change)

CriticalEffort: M · ~2–4 days
Issue
Content is aggregated per-visitor in the browser via free third-party proxies. Each first visit triggers 80+ requests; two visitors see different "briefings"; allorigins/corsproxy/rss2json can rate-limit or vanish at any time.
Impact
Slow content paint, inconsistent product, fragile dependencies, visitor IPs leaked to third parties, and zero indexable content — this one design decision is the root cause of most performance, SEO and reliability findings below.
Fix
A Cloudflare Worker Cron Trigger (every 15–30 min) fetches all feeds server-side, applies the existing filters (reuse the engine code — it ports cleanly), and writes one JSON blob to KV (feed:latest). Clients fetch a single cached /feed-data endpoint. The same blob then powers /feed.xml, send-digest, the archive, and server-rendered HTML (R19). Stays within Cloudflare's free tier.
R2

One source-of-truth for feed sources

HighEffort: S · hours
Issue
Four divergent source lists: pai-feed-engine.js, functions/feed.js, functions/send-digest.js, and the rss-proxy allow-list. They have already drifted (The Verge is digest-only; several allow-listed domains are no longer fetched).
Impact
Silent breakage: adding a source in one place but not the proxy allow-list fails quietly; the digest and the site show different worlds.
Fix
A single shared sources.js module (url, src key, label, color, domain, category) imported by all functions and used to generate the allow-list. Falls out naturally if R1 is done.
R3

KV read-modify-write races & the single-key archive

MediumEffort: M
Issue
Counters in stats.js and the merge in archive-data.js are read-then-write on eventually-consistent KV — concurrent requests lose increments and archive items. countDays() also lists an unbounded key prefix on every /stats request. The whole archive lives in one KV value (5,000-item hard cap, "papers kept forever").
Impact
Counters silently undercount as traffic grows; /stats latency grows with history; archive writes can clobber each other; the forever-retention promise will hit the cap.
Fix
Short term: cache the days count, accept counter drift. Right answer: migrate archive + stats to Cloudflare D1 (free SQLite). D1 gives atomic increments, real queries for archive search/filtering, removes the 5,000 cap — and becomes the foundation for per-article pages (R21).
R4

Remove dead artifacts shipping to production

HighEffort: S · 1 hour
Issue
index_classic_backup.html (198 KB) and promptai_google.html deploy to production as publicly crawlable URLs; a static sitemap.xml coexists with the dynamic /sitemap function it's redirected to; emails/*.html previews show a retired dark design.
Impact
Duplicate-content risk for SEO, confusing crawl surface, and exactly the "drifting copies" problem CLAUDE.md warns about — inside the repo itself.
Fix
Move backups out of the deployed branch (or a /_archive folder excluded via .gitignore/build config), delete the static sitemap.xml so only the dynamic one serves, and regenerate email previews from digestHtml().
R5

Minimal CI gate before auto-deploy

MediumEffort: S
Issue
Every push to main deploys straight to production with no checks — no HTML validation, no lint, no smoke test.
Impact
A single bad commit takes the live site down until manually noticed (there is no monitoring either — R9/R40).
Fix
GitHub Action on push: HTML validation + ESLint + a deploy-preview smoke test that curls /, /stats, /feed.xml, /sitemap.xml and fails on non-200. Cloudflare Pages preview deployments make this nearly free.

2 · Performance, scalability & reliability

End-user and operator perspective
R6

Cut the first-visit network storm

HighSolved by R1
Issue
An uncached visit issues ~80+ requests (9 news feeds × up to 4 proxy racers, 8 blog feeds, arXiv, 60 individual HN item fetches) before the briefing fills in. Skeletons mask it, but on mobile data this is seconds of spinner and real battery/data cost.
Impact
Slowest part of the perceived experience; the #1 thing a first-time mobile visitor feels.
Fix
With R1, first paint needs exactly one JSON fetch (edge-cached). Target: briefing content visible < 1s on 4G.
R7

Compress og-image.png (565 KB)

LowEffort: S · minutes
Issue
The social-share image is 565 KB — large for a 1200×630 card.
Impact
Slow link previews on WhatsApp/X/LinkedIn; some platforms time out and show no image.
Fix
Re-export ≤ 100 KB (optimized PNG or JPEG; keep PNG if it has flat brand colors).
R8

Self-host fonts or add preloads

LowEffort: S
Issue
Roboto + Roboto Serif load render-blocking from Google Fonts (two CSS round-trips before text settles); display=swap is set, which helps.
Impact
Minor layout shift and slower LCP, especially on first visit from India/high-latency networks.
Fix
Self-host the two families as WOFF2 on Pages (also removes a third-party dependency), or at minimum preload the two stylesheet URLs.
R9

Uptime + feed-health monitoring

HighEffort: S
Issue
No alerting exists. A dead source, an expired Resend key, or a failed deploy is only discovered by visiting the site. CHANGES.md already flags "log feeds that return 0 items" as pending.
Impact
The weekly digest could silently no-op for weeks (it already returns 200 on "all feeds failed"); subscriber trust is hard to win back.
Fix
(a) Free uptime monitor (UptimeRobot / Cloudflare Health Checks) on / and /feed.xml. (b) In the digest run, write a lastRun report to KV (per-feed item counts, send count, errors) and expose it at an authenticated /health. (c) Cache the last good digest so a total feed outage sends last week's "best of" rather than nothing.
R10

PWA: offline shell + installability

MediumEffort: M
Issue
No manifest or service worker; the localStorage cache already holds content, but the shell itself needs a network.
Impact
News-reading happens on commutes; an installable, offline-capable briefing meaningfully raises return-visit frequency on mobile.
Fix
Add manifest.json + a small service worker (cache-first shell, network-first /feed-data). Pairs naturally with R1's single-fetch model.

3 · Security & privacy

The account system and email list are the crown jewels
R11

No password reset flow

CriticalEffort: M · 1–2 days
Issue
/auth supports signup/login/session/logout only. A forgotten password permanently locks the user out (and "sign up again" fails with "account already exists").
Impact
Guaranteed, unrecoverable churn for every forgotten password; support burden lands on you personally.
Fix
Add action:'reset-request' (emails a 30-min single-use token via Resend — the pipe already exists) and action:'reset-confirm'. ~80 lines in auth.js + a small sheet UI.
R12

No rate limiting or lockout on /auth

CriticalEffort: S · hours
Issue
subscribe.js has a tidy KV rate-limiter; auth.js has none. Unlimited login attempts enable credential-stuffing and brute force; unlimited signups enable KV pollution.
Impact
Account takeover risk and abuse cost; with PBKDF2 at 100k iterations, each guess is also free compute you pay for.
Fix
Port the same KV limiter (e.g. 10 attempts/IP/15 min + per-account exponential backoff). While there, raise PBKDF2 to 600k iterations (current OWASP guidance for SHA-256) for new hashes, upgrading old ones on successful login.
R13

Tighten CORS and session storage

HighEffort: S
Issue
Every function responds Access-Control-Allow-Origin: * — including /auth. Session tokens live in localStorage, readable by any script that ever gets injected.
Impact
Wider attack surface than needed; an XSS anywhere becomes full account compromise.
Fix
Lock CORS to https://promptai.in on auth/data endpoints (same-origin callers don't need *). Medium-term: move sessions to HttpOnly; Secure; SameSite=Lax cookies.
R14

Cron secret travels in the URL

MediumEffort: S
Issue
/send-digest?key=CRON_SECRET puts the secret in query strings, which land in logs, browser history and any intermediary.
Impact
A leaked key lets anyone spam your entire subscriber list from your domain.
Fix
Accept the key via Authorization: Bearer header (keep query-param support briefly for migration), or move to a real Worker Cron Trigger (R1) and drop the public endpoint entirely.
R15

No security headers

HighEffort: S · hours
Issue
No _headers file: missing Content-Security-Policy, X-Frame-Options, Referrer-Policy, HSTS, Permissions-Policy.
Impact
The site renders third-party RSS content; CSP is the safety net if any escaping bug slips through. Clickjacking and referrer leakage are free to fix.
Fix
Add _headers with HSTS, X-Frame-Options: DENY, Referrer-Policy: strict-origin-when-cross-origin, and a CSP in Report-Only first (the inline onclick handlers need 'unsafe-inline' initially — note it as debt to refactor).
R16

No privacy policy or terms

CriticalEffort: S · half-day
Issue
The platform stores emails, names, password hashes and reading history, and shares data with processors (Resend, Cloudflare, Google Fonts/Translate, image proxy) — with no privacy policy, no terms, anywhere.
Impact
Non-compliance with GDPR and India's DPDP Act 2023; blocks future ad networks and sponsor deals; erodes the "independent, trustworthy" brand the About panel claims.
Fix
Ship privacy.html + terms.html (what's collected, why, processors, retention, deletion contact), link from the About panel, sign-up sheet, and every email footer. Add an account-deletion path (action:'delete-account').
R17

Unsubscribe accepts a bare email parameter

LowEffort: S
Issue
/unsubscribe?email=… lets anyone unsubscribe anyone whose address they know.
Impact
Low-grade abuse vector; a rival or prankster can silently shrink your list.
Fix
Sign the link: ?email=…&sig=HMAC(email, secret) — ~15 lines, no stored state.
R18

No double opt-in on subscribe

HighEffort: M · 1 day
Issue
Any typed email is added to KV and immediately mailed a full briefing. Typos, pranks and bot entries become permanent list members.
Impact
Spam-trap hits and complaints damage sender reputation — compounding with unverified domain auth (R37) this is what gets briefings sent to spam. Also a consent requirement under GDPR.
Fix
Store new emails as pending, send a one-click confirm link (HMAC-signed, reusing the R17 helper), flip to active on click. Digest sends only to active.

4 · SEO & internet visibility

The meta layer is excellent; the content layer is invisible
R19

Server-render real headlines into the HTML

CriticalEffort: M · with R1
Issue
Canonicals, OG cards, JSON-LD, robots and dynamic sitemap are all done well — but they decorate three near-empty shells. Every story, paper and blog arrives via JS from third-party APIs, which Google rarely indexes as your content (DISCOVERABILITY.md already flags this).
Impact
PromptAI can rank for its brand name and little else. All the freshness signals (changefreq: hourly) point at content Googlebot can't see.
Fix
Once R1's feed JSON exists in KV, add a Pages middleware (or scheduled HTML regeneration) that injects the current top ~20 stories as static HTML into index.html (the JS then hydrates over it). Googlebot finally sees real, hourly-fresh content.
R20

No <h1> and weak heading hierarchy

MediumEffort: S · hours
Issue
Section titles ("Your briefing") are styled <div>s; pages have h3s with no h1/h2 above them.
Impact
Search engines and screen readers both lose the page outline — a two-for-one fix with accessibility.
Fix
Make the active view's title an <h1> (visually identical), section dividers <h2>, card titles <h3>.
R21

Create rankable URLs: topics, guides, issues

HighEffort: L · iterative
Issue
The sitemap contains exactly 3 URLs. There is nothing for long-tail queries ("best free AI courses 2026", "AI agents news") to land on.
Impact
Organic growth is structurally capped at ~3 keywords; every competitor newsletter (TLDR AI, Ben's Bites, The Rundown) ranks via issue archives and topic pages.
Fix
In priority order: (a) split Learn AI into per-guide static pages (/learn/prompt-engineering.html …) — stable, unique, already written; (b) publish each weekly digest as a static issue page (/briefing/2026-06-09.html) — auto-generated from digestHtml(), doubles as the "sample issue" (R32); (c) topic hub pages fed from the D1 archive (R3).
R22

Distribution & backlinks (outside the codebase)

HighEffort: ongoing
Issue
The DISCOVERABILITY.md checklist (Search Console, Bing, rich-results tests, directory listings) appears not yet executed; there is no social auto-posting despite /feed.xml being built for it.
Impact
Backlinks move rankings more than any on-page change; right now nothing builds them.
Fix
Execute the existing checklist (GSC + Bing, ~30 min); wire feed.xml to Buffer/Zapier for auto-posting to X/LinkedIn; submit to AI directories and newsletter aggregators. Bing also feeds ChatGPT/Copilot answers — cheap AI-search visibility.
R23

Noindex what shouldn't rank

MediumEffort: S
Issue
The classic backup and prototype pages are crawlable (overlaps R4); search-result and archive query URLs aren't excluded beyond robots.txt.
Impact
Duplicate/near-duplicate pages dilute the small site's crawl signal.
Fix
Delete or noindex backups; keep canonical discipline on parameterized URLs.

5 · Accessibility & usability

Reduced-motion and focus styles are done; interaction semantics are not
R24

Story cards are unreachable by keyboard

HighEffort: S · hours
Issue
Cards are <article onclick=…> with no link, tabindex or key handler. A keyboard or screen-reader user cannot open a single story — the core action of the product.
Impact
WCAG 2.1.1 failure on the primary interaction; excludes assistive-tech users entirely.
Fix
Make each card title a real <a href> (best: also gives crawlable links once R19 lands), with the modal intercepting click; or add role="link" tabindex="0" + Enter/Space handling in storyCard().
R25

Modal & sheet focus management

MediumEffort: S
Issue
The article modal and sign-in sheet don't trap focus, set aria-modal, or return focus on close.
Impact
Keyboard users tab into the dimmed page behind the dialog; screen readers don't announce the context switch.
Fix
Use a shared ~30-line dialog helper: role="dialog" aria-modal="true", focus trap, Esc to close, restore focus to the opener.
R26

Tab bar semantics

LowEffort: S
Issue
The Home/News/Research/Blogs/Tools switcher is visually tabs but semantically plain buttons mixed with links; no aria-current/aria-selected.
Impact
Screen readers can't tell which section is active.
Fix
Add aria-current="page" on the active tab (simplest, honest for this hybrid pattern).
R27

Views don't update the URL

MediumEffort: S
Issue
Switching tabs or opening search changes nothing in the address bar: Back exits the site, sections can't be linked or bookmarked, and analytics will see one page.
Impact
Surprising back-button behavior is a top usability irritant; "send someone the Research tab" is impossible.
Fix
history.pushState with #news/?view=news in go(), restore on load and popstate (the cross-page search handoff shows the pattern is already half-built).

6 · Product, UX, AI capability & differentiation

Product-owner perspective: what makes this more than an RSS reader?
The strategic gap: PromptAI currently competes with TLDR AI, Ben's Bites, The Rundown and smol.ai on curation alone — with no AI in the product, no prompts despite the name, and no per-user value beyond reading history. R30 + R31 are the two changes that create a defensible identity; everything else polishes the funnel around them.
R30

Add real AI: daily TL;DR + "one paper, explained"

Critical (differentiation)Effort: M · 2–3 days
Issue
The welcome email literally promises "One paper, explained — in plain English", but the digest ships the paper's raw abstract. The site offers no summaries, explanations or synthesis — the "AI Research Hub" contains no AI.
Impact
Broken product promise + missed differentiation. AI-generated synthesis is exactly what readers of competing briefings pay attention (and money) for.
Fix
In the aggregation worker (R1), one batched LLM call per cycle — Workers AI (free tier) or the Claude API — producing: a 5-bullet "Today in 60 seconds" for the homepage lead, a plain-English explainer for the day's top paper, and the digest's paper section. Cached in KV; cost is cents/day; clearly labeled as AI-generated with a link to the original.
R31

Ship a prompt library — own the name "PromptAI"

HighEffort: M · iterative
Issue
The domain promises prompts; the closest the product gets is 4 external links in the education page's prompt-engineering section.
Impact
"AI prompt" queries are an enormous, durable search market that the brand name is pre-positioned for; each prompt is also a static, rankable URL (compounds R21).
Fix
A curated prompt library: categories (writing, coding, research, learning), one page per prompt with copy button, "works with Claude / ChatGPT / Gemini" notes, and a weekly "prompt of the week" slot in the digest. Start with 20 hand-written prompts in a JSON data file; quality over volume.
R28

Surface save / like — the backend already exists

HighEffort: M · 1–2 days
Issue
auth.js and pai-account.js already store and sync saved/liked lists (capped, merge-synced, cross-device) — but no button in the UI ever writes to them. (Already item 4 on NEXT_SESSION.)
Impact
Sign-in currently offers almost no value ("recently read" only), so nobody signs in, so retention features have no audience. Bookmarking is the cheapest reason-to-return available.
Fix
Bookmark icon on cards + modal; "Saved" section in the account sheet; "3 stories you saved this week" block in the digest for signed-in subscribers.
R29

Add a dark theme

MediumEffort: S · half-day
Issue
Three appearance options — all light. The token system makes dark a ~30-line addition, plus a prefers-color-scheme default.
Impact
Dark mode is the single most expected appearance option for a reading product, especially for the developer audience this site targets.
Fix
One [data-theme="dark"] token block + an "Auto (system)" option in the appearance menu. Audit the hard-coded pill/access colors while there.
R32

Conversion: social proof + a sample issue

MediumEffort: S
Issue
Subscribe forms ask for trust without evidence: no subscriber count, no testimonial, no way to see what the briefing looks like before handing over an email.
Impact
"Show me first" is the #1 objection for newsletter signups; a sample issue routinely lifts conversion double-digits.
Fix
"Read a sample briefing →" link under every subscribe form, pointing at the latest static issue page (R21b). Add a subscriber count once it's a number you're proud of.
R33

Rethink the public "Live stats" panel

LowEffort: S
Issue
"Total visitors / Articles read" are operator metrics shown to users; while numbers are small they undermine rather than build credibility, and the counters are knowably lossy (R3).
Impact
A small-number stat panel signals "small site" on every page view.
Fix
Replace with content stats that are impressive at any scale: "142 stories curated this week · 20+ sources · updated 12 min ago". Keep visitor counts in a private dashboard (R40).

7 · Content organization & information architecture

R34

"Blogs" vs "Learn AI" naming collision

MediumEffort: S
Issue
The tab labeled Blogs renders a view internally called learn ("Worth learning"), while a separate page is called Learn AI. Users can't predict which holds what; the code's naming reflects the same confusion.
Impact
Navigation hesitation on every visit; misfiled mental model of the site.
Fix
Rename the tab to Deep dives (or "Explainers") and keep Learn AI for courses/paths. Align the internal view key when convenient.
R35

Tools directory: hard-coded and aging

MediumEffort: S
Issue
16 tools live as a constant inside pai-google-ui.js; entries like "Midjourney v6" and "DALL·E 3" already read dated in mid-2026. No "last reviewed" signal, no way for users to suggest tools.
Impact
A stale directory quietly contradicts the "hands-on, honest reviews" brand claim.
Fix
Move to tools.json with lastReviewed dates rendered on cards; quarterly review pass; optional "suggest a tool" mailto/form. (Also unlocks affiliate tagging — R43.)
R36

Archive retention promise vs the 5,000-item cap

LowWith R3
Issue
Archive policy says papers/blogs are "kept forever", but everything lives in one KV value hard-capped at 5,000 items — forever ends silently.
Impact
Future data loss that contradicts a stated guarantee.
Fix
Covered by the D1 migration in R3; until then, raise visibility by logging when the cap trims items.

8 · Newsletter — the core product

Most of the hard deliverability work is already in code
R37

Verify SPF / DKIM / DMARC for promptai.in

CriticalEffort: S · 30 min DNS
Issue
CHANGES.md flags domain auth as unverified in Resend. List-Unsubscribe headers and plain-text parts (already shipped — well done) only help alongside verified auth.
Impact
This is the single biggest factor in whether the briefing lands in inboxes or spam. Everything else about the newsletter is downstream of it.
Fix
Add Resend's SPF/DKIM DNS records at Cloudflare, confirm "Verified", then publish a DMARC record (p=none with rua reporting to start). Send Gmail/Outlook/Yahoo test passes after.
R38

One canonical email template + honest previews

MediumEffort: S
Issue
The digest was restyled to the light briefing brand, but emails/*.html previews still show the retired dark design (NEXT_SESSION item 3), and the welcome email uses its own third style.
Impact
Brand inconsistency at the most important touchpoint; previews that lie make future edits error-prone.
Fix
Generate the static previews from digestHtml() itself; restyle welcomeHtml() with the same tokens.
R39

Build the newsletter growth loop

MediumEffort: M
Issue
The only acquisition channel for the digest is the site itself; issues vanish after sending.
Impact
No compounding: every issue's content is consumed once and discarded.
Fix
(a) Publish every issue as a web page (R21b) and link "Past briefings" from the rail; (b) add a "Forwarded this? Subscribe here" footer line; (c) auto-share each issue to X/LinkedIn via feed.xml (R22).

9 · Analytics, observability & operational excellence

R40

Install real (privacy-friendly) analytics

HighEffort: S · 1 hour
Issue
The KV counters answer "how many" but never "from where, doing what, converting at what rate". There is no funnel visibility at all: traffic sources, subscribe conversion, tab usage, story CTR, returning-visitor rate are all unknown.
Impact
Every growth and monetization decision in this report is currently a guess; sponsors will ask for numbers you can't produce.
Fix
Cloudflare Web Analytics (free, cookieless, one script — no consent banner needed, consistent with the privacy posture). Define 4 KPIs: visit→subscribe rate, D7 return rate, story CTR, digest open rate (Resend webhooks → a small KV log).
R41

Error visibility & a /health endpoint

MediumEffort: S
Issue
Client-side failures (a proxy dying, a feed gone 403, JS exceptions) are swallowed by design — silent resilience is good UX but means nobody learns the site degraded. SOURCE_STATUS exists but is never reported anywhere.
Impact
"News tab feels emptier lately" is currently undiagnosable.
Fix
Authenticated /health page: per-source last-success timestamps (from the aggregation worker after R1), digest last-run report (R9), KV binding checks. Optional: a tiny client error beacon counting JS exceptions per page.

10 · Monetization & commercial viability

Constraint to honor: the site promises "Free and open — no paywalls, ever"
R42

Newsletter sponsorship (primary path)

MediumEffort: S when ready
Issue
No revenue model exists. The cleanest one for this product — a single clearly-labeled sponsor slot in the digest — is standard for AI briefings and doesn't violate the no-paywall promise.
Impact
At newsletter scale (1k+ engaged subscribers), one weekly classified-style slot is realistic early revenue and forces no product compromise.
Fix
Prereqs first: analytics (R40), domain auth (R37), double opt-in (R18) and issue archive (R21b) — sponsors check all four. Then a one-page media kit (audience, open rate, geography) and a sponsor field in the digest builder.
R43

Disclosed affiliate links in the tools directory

LowEffort: S
Issue
The tools directory sends free referral traffic to 16 commercial products.
Impact
Modest passive revenue with zero UX change — if disclosure is honest, it also doesn't dent the independent-review claim.
Fix
